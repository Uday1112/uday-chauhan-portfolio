PROJECT 01 · AUTOMATION
Reporting Workflow Automation
4 hrs saved weeklySame-day delivery6 workflows rebuilt
THE PROBLEM

When I joined Variance InfoTech there were 6 recurring reports that an analyst was producing manually every week. Each one involved opening source files, copying data into Excel, running formulas, formatting, and emailing outputs. The process was eating up roughly 4 hours of analyst time weekly and producing inconsistent results because different people handled it differently depending on who was available.

WHAT I BUILT
  • Analysed each of the 6 workflows individually to understand the data sources, transformation logic, and output format required
  • Wrote Python scripts using pandas for each workflow that pulled data from SQL Server, applied all transformation and cleaning logic, and generated formatted output files automatically
  • Set up SQL Server scheduled jobs to trigger each Python script at the right time overnight so outputs were ready by 7am
  • Built error handling and logging into each script so failures would surface immediately rather than silently producing wrong outputs
  • Documented each workflow so any team member could maintain or modify it without asking me

PROJECT 02 · REPORTING
Power BI Dashboard Consolidation
14 reports to 812 daily usersData model restructured
THE PROBLEM

I inherited a Power BI environment with 14 overlapping reports built by multiple contributors over 2 years. The data model had grown inconsistent, DAX measures were duplicated across reports with slightly different logic, and stakeholders were asking clarifying questions about the numbers every week because different reports showed different figures for the same metrics. Nobody fully trusted what they were looking at.

WHAT I BUILT
  • Audited all 14 reports to understand which were actively used, which were duplicates, and which could be safely retired
  • Restructured the underlying data model following proper star schema principles with clear fact and dimension tables
  • Rewrote all DAX measures with consistent logic and naming conventions so the same metric always shows the same number regardless of which report surfaces it
  • Consolidated 14 reports into 8 well structured ones covering all the same use cases without the redundancy
  • Presented the consolidated suite to 12 regular users across operations and finance and incorporated their feedback before final release

PROJECT 04 · DATA PIPELINE
Multi-Source Data Integration Pipeline
5 systems connected3 people replacedConsistent daily output
THE PROBLEM

Data needed for weekly reporting lived in 5 separate systems: SQL Server tables, Excel files, a cloud CRM, and two external data feeds. Three people were manually pulling from each source, reconciling discrepancies, and combining the outputs every week. The process took hours and produced slightly different results depending on who ran it and in what order they pulled the sources.

WHAT I BUILT
  • Mapped all 5 source systems, understanding the data structure, update frequency, and business logic needed for each
  • Built a Python ETL pipeline that pulled from all 5 sources automatically, applied transformation and reconciliation logic, and loaded clean output into a single SQL Server destination table
  • Added data validation checks at each stage to catch missing values, duplicates, or out of range figures before they reached the output
  • Scheduled the pipeline to run nightly so the combined dataset was ready every morning without any manual intervention
  • Documented the pipeline architecture and transformation logic so it could be maintained or extended by any technical team member

PROJECT 06 · DATA QUALITY
Anomaly Detection and Root Cause Fix
8% undercount identifiedRoot cause tracedProtocol changed
THE PROBLEM

During routine data quality review of weekly compliance figures for the Google retail program, I noticed that one product category was consistently showing lower compliance numbers than field reports suggested. The gap was small enough that it had not triggered any alerts, but it was appearing week after week in the same direction, which suggested a systematic issue rather than random variation.

WHAT I BUILT
  • Ran SQL and Python queries to isolate the affected product category and compare figures across all data sources feeding into the compliance metric
  • Traced the discrepancy back to a logic error in the field data collection form that was causing a specific type of compliance action to be recorded in the wrong category
  • Documented the issue with specific examples, affected record counts, and the business impact of the undercount on how the region was evaluating that category
  • Worked with the technical team to correct the collection logic and validated the fix against historical records
  • Presented findings to client stakeholders in a format that explained both the technical cause and the business implication clearly
